Operation Fanout

Applying NLA's verification discipline to multi-agent pipelines

The Anthropic Natural Language Autoencoders paper proves something more general than its title suggests: LLM verbalisation has a signature failure mode. Themes faithful, specifics drift, true claims recur, false claims do not. If that signature is universal to verbalisation — and there is no reason it should be limited to activation-to-text — the same heuristics work on any LLM output. Fanout is the engineering test of that claim.

A direct translation of NLA reading discipline into a working multi-agent CLI

I.The problem with multi-agent at scale

Run four LLM agents in parallel. Three return real work. One returns a report shaped exactly like real work — same headings, same severity tags, same imperative tone — that on close reading turns out to cite a file that does not exist, or invents a line number, or recommends a library nobody wrote. The reducer ingests all four. The user reads the synthesis, acts on a finding that was hallucinated, and learns nothing about which worker invented it.

This is not a bug in any one model. It is a property of how language models verbalise complex internal state under pressure. Detecting the property is the work; once you detect it the rest of the engineering is straightforward.

Five sessions running in parallel is a fast queue. Five sessions whose output you can trust is a method.

II.What the NLA paper actually proved

Anthropic's Natural Language Autoencoders (May 2026) trained an autoencoder pair: a verbaliser that turns model activations into text, a reconstructor that turns the text back into activations. The optimisation target is reconstruction loss — nothing in the loss explicitly rewards readability or honesty. What emerges is the interesting part. Five empirical findings, in order of how load-bearing each is for fanout:

  1. Themes are faithful; specifics drift. Verbalisations correctly name the shape of what an activation encodes (Korean dynastic records, security audit, planning a rhyme). They incorrectly name specific entities and details that fit the shape (a king who was not mentioned, a file that does not exist, a line number invented to look concrete).
  2. True claims recur across token positions; false claims do not. A claim that appears in the verbalisation at multiple adjacent tokens is more likely to be supported by the underlying activation. Singletons should be read with suspicion.
  3. The reconstructor is a partial verifier. Removing a true claim from a verbalisation hurts reconstruction loss more than removing a false one. The text you can take out without breaking the round-trip is text the activation did not really need.
  4. Confabulation rate is roughly flat across training. More RL does not fix it. The signature is not an artefact of an under-trained verbaliser; it is a property of the verbalisation regime itself.
  5. Reading discipline beats fine-tuning. Anthropic's own practice: "read NLA explanations for the themes they surface rather than for single claims, and corroborate findings with independent methods before fully trusting them." NLA-equipped auditing agents identified the root cause of an intentionally misaligned model without training-data access — by reading themes and corroborating.

The paper's framing is "we built a tool for interpretability researchers to read activations." The framing one level up is: this is what LLM verbalisation looks like when you can measure it against a known signal. The signal is reconstruction loss in the paper. In product code, the signal is whether the verbalisation grounds in observable reality (file content, repo state, runtime behaviour).

III.The innovation, in three claims

Fanout is not "uses NLAs." It is the claim that NLA's empirical findings describe a property of LLM verbalisation that survives outside the autoencoder regime — and the engineering payoff of taking that claim seriously inside a multi-agent CLI. Three concrete claims, each falsifiable:

Claim 1 — Universal signature

The four findings above describe LLM verbalisation in general, not the autoencoder pipeline specifically. A worker producing a markdown audit report is verbalising its task model. Themes will be faithful. Specifics will drift. True claims will repeat. False claims will not. The paper measured this with reconstruction MSE. We measure it with file:line ground-truth checks against the bundle the planner already supplied. Falsifiable: if confabulated specifics in worker output do not cluster on absent paths and out-of-range lines, the claim fails.

Claim 2 — Cross-agent recurrence is a stronger signal than within-token recurrence

The paper measures recurrence within a single verbalisation (same claim across adjacent tokens). Multi-agent pipelines provide a second axis: independent workers, run on the same plan, raising the same claim. Worker independence is enforced at the validator (no worker reads another's output, no shared read_files overlap beyond one for by_file strategy). Cross-worker recurrence is therefore information not available to a single-agent NLA — it is corroboration in the strict statistical sense. Falsifiable: if cross-worker-recurring claims are not measurably more accurate than singletons, the claim fails.

Claim 3 — Filter at the API boundary, not in the model

The paper observes that NLA RL does not reduce confabulation rate over training. The architectural lesson: don't try to make the verbaliser honest; build the verifier as load-bearing infrastructure around it. Fanout puts the verifier between worker dispatch and reducer, where the bundle is still in scope. The reducer never sees raw worker output; it sees a LensReport with claims pre-bucketed into trust bands. Falsifiable: if filtered output is not measurably more useful than unfiltered output on the same task, the claim fails.

Each claim is the kind of statement a research paper would defend with an evaluation. We defend them with a working CLI and an open repo: github.com/muhibwqr/fanout. The fixtures intentionally include confabulated worker outputs so the test suite verifies the filter's behaviour against the failure mode the paper documents.

NLA's finding is that LLM verbalisation has a signature. Fanout's bet is that the signature is portable. The repo is the test.

IV.Architecture

The pipeline is four stages on the unfiltered path, six with the lens layer engaged. Workers are claude -p subprocesses dispatched in parallel via asyncio.gather (headless) or visible tmux panes. Workers do not communicate. The validator enforces independence at plan time; the lens enforces verification at output time.

The lens layer is the new component. Each of its eight stages implements one of the paper's findings (table next section). The auditor reducer never sees raw worker output — it sees the LensReport, which is the paper's "themes versus specifics" reading discipline rendered as a data structure.

V.The lens, stage by stage

Eight stages. Each stage exists because the paper supplied a finding that requires it. Removing any stage degrades the filter against a specific failure mode the paper documented.

1Extract claims. Claude judge per worker output emits an array of atomic factual claims with cited paths and lines.Mirrors the Haiku 4.5 judge the paper used to characterise confabulations.Aggregate scoring requires atomic units. Whole-document trust is too coarse.
2Bucket specificity as theme / entity / detail."Thematic claims are more accurate than specific ones."The paper's strongest finding for product use. Themes default to trust; specifics default to verify.
3Ground-check cited paths and line numbers against the bundle the planner saw.Confabulation characterisation: false specifics anchor to true themes.The cheapest stage. Pure-Python, no LLM call. Catches "src/auth/oauth.py" when no such file exists in the bundle.

Jaccard token-set match, threshold 0.4."True claims recur across token positions."Cheapest reliability signal. Cross-worker recurrence is the multi-agent generalisation the paper does not reach.
5Reconstruction. One Claude call per claim: "given the cited file content, is this claim supported, partially supported, or unsupported?" Strict on details, lenient on themes."AR partially distinguishes true from false claims" — removing true claims hurts reconstruction more than removing false ones.The most expensive stage. Disable with --lens-fast. The closest analog to the AR's role in the original autoencoder.
6Score each claim via a rule table → high / med / low trust + rationale.Synthesises stages 2–5.Single trust value per claim; rationale string explains the assignment so the user can audit the auditor.
7Score each worker. fake_success_score = 1 − (high-trust-specifics / total claims). The auditor reducer's output

The reducer receives the LensReport — never raw worker output — and emits a four-section markdown report. Sections mirror the paper's discipline directly.

# Findings (high trust)
Lead with thematic claims that reconstruction supports + grounded entities.
Cite W{i}.

# Details (verified)
Specific file:line claims that ground-truth-checked AND reconstruction-supported.
Act on these without re-checking. Cite W{i} with path:line.

# Suspect — verify before acting
Low-trust claims, kept but flagged with the failure mode:
  - "no such file"               (path absent from bundle)
  - "line out of range"          (line beyond excerpt)
  - "no support in cited content" (reconstruction unsupported)
Paper says false claims are often *thematically related* to the truth and
may seed real findings. Don't delete them. Don't trust them either.

# Worker reliability
W1 — 6 high, 2 med, 1 low. fake_success_score=0.18.   reliable
W2 — 1 high, 1 med, 4 low. fake_success_score=0.83.   flagged, retried
W3 — 5 high, 3 med, 0 low. fake_success_score=0.27.
W4 — 0 high, 0 med, 0 low. fake_success_score=1.00.   retry produced nothing

The Suspect section is the part most people initially want to delete. The paper's data argues against deletion: false claims tend to be thematically related to the truth and sometimes seed real findings on follow-up. Surface them with their failure mode; do not silently incorporate them into the synthesis.

VIII.Honest limits



The translation from research finding to product filter is incomplete in three ways. Naming each is part of the work.



    
  
  • No real activations. Claude is closed; we cannot run a verbaliser on its activations. The lens applies the paper's heuristics to worker output, which is the closest observable analog. The paper's claims about activations are stronger than ours about text — we inherit only the heuristics that survive the abstraction. Real-NLA integration via Neuronpedia or Hugging Face would require running workers on Qwen/Gemma/Llama in parallel; deferred.
    • 
  
  • Cross-agent recurrence is informally measured. Jaccard token-set with threshold 0.4 is a crude proxy for "same claim." Embedding-based matching would be tighter but adds a dependency. v3 territory.
    • 
  
  • Tokens are not free. Lens mode adds claim-extraction (1 call per worker) plus reconstruction (1 call per claim) plus the auditor reducer. For a four-worker, twelve-claim audit, lens roughly doubles the LLM call count. The argument is not "fewer tokens" but "tokens spent on verified signal rather than mediocre confabulation." Where the user prefers raw tokens and serial reading, omit --lens.
    •
