I.The fundamental error in workstation

Workstation is a 40 KB shell script, top to bottom. You run sh startup.sh and it installs 35+ tools in a fixed order. It works. It is also a category of mistake the field has been repeating for fifteen years.

Seven concrete failures of the imperative one-shot pattern:

  1. Not idempotent. Re-running re-installs. brew install X on an installed formula sometimes upgrades, sometimes no-ops, sometimes errors. Output depends on the tool. The script has no model of this.
  2. No plan before apply. Want to see what will happen before it does? Read 40 KB of bash. There is no preview. You trust or you don't.
  3. No rollback. Step 43 fails on a stale cask lock. Steps 1–42 already ran. Step 44–N never will. You have a half-configured machine and no way back.
  4. No state. Three months later, did workstation install node, or did you? The script has no record. You cannot diff what's mine vs what came from the tool.
  5. No selection. Want only AWS + Node? Fork the script and delete lines. No profiles. All-or-nothing.
  6. No verification. Script exits. Did docker ps actually work? Did aws --version succeed? Unknown until you try to use the tool.
  7. No drift detection. You install something off-script. Workstation has no idea. Re-running can clobber your custom state.

Each of these is a Terraform/Nix lesson. Mainstream "set up your dev env" scripts ignored them for a decade. fanout adopts them all.

The point isn't that workstation is bad. The point is that imperative bootstrap scripts as a category were a mistake the field repeated. fanout is the answer.

II.The fix, in five commands

fanout init                    # writes ~/.fanout/workstation.yml from a template
fanout plan                    # shows + (install) / - (remove) / = (keep) per bucket
fanout apply                   # converges your machine; snapshots before; idempotent
fanout state diff              # detects drift since last apply
fanout rollback                # restores the most recent snapshot

Plus the AI shortcut:

fanout ai "set up a Python ML rig with PyTorch, Jupyter, and Docker"
# Claude writes the manifest. You see it. [a]ccept / [e]dit / [r]egen / [q]uit. fanout apply.

III.Architecture

One YAML file is the source of truth. The engine reads it, asks each adapter what's currently installed, computes a diff, applies it. State written to ~/.fanout/state.json. Snapshots written to ~/.fanout/snapshots/<ts>.json before each apply.

fanout claude "task a" "task b" [--no-tmux] [--keep-tmux]

The claude subcommand is a bonus: dispatch N user-supplied tasks to N parallel claude -p sessions in tmux panes. Same dispatch engine fanout uses for parallel installers under apply.

VI.Manifest format

version: 1

profiles:
  default:   [base, web, cloud]
  python-ml: [base, python, ml]

modules:
  base:
    brew: [git, jq, ripgrep, fzf]
  web:
    brew: [node]
    cask: [visual-studio-code]
    npm_global: [typescript, yarn]
  cloud:
    brew: [awscli, kubectl]
    cask: [docker]
  python:
    brew: [python]
    pip: [poetry, ruff]
    curl:
      - name: nvm
        marker: "~/.nvm/nvm.sh"
        install: "curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash"

settings:
  apply:
    parallelism: 4
    snapshot_before: true
  verify:
    checks:
      - cmd: "git --version"
      - cmd: "node --version"

Profiles compose modules. Modules group tools into buckets by adapter (brew, cask, npm_global, pip, curl). Curl items declare a marker path that signals already-installed — that's how third-party shell installers (NVM, Oh-My-Zsh, Docker Compose) get idempotence.

VII.AI mode

The fanout ai subcommand sends the user's prose + the current manifest to Claude with a tight system prompt that says: output ONLY YAML matching this schema. The result is validated against the same schema validator the manifest loader uses (aggregated errors, no silent failures). User sees the generated YAML, picks [a]ccept / [e]dit / [r]egen / [q]uit. On accept, the manifest is written. Then fanout apply like usual.

This is not magic. It is one Claude call wrapped in a validator. The validator is load-bearing: Claude can and does emit specifics that don't match valid Homebrew formula names. The schema catches it before you run a brew install on a typo.

VIII.HTML reports for every command

Inspired by Thariq's Unreasonable Effectiveness of HTML: every subcommand emits a self-contained HTML artifact, written to ~/.fanout/reports/<timestamp>-<cmd>.html, and auto-opens in your browser on macOS. Markdown is for diffs. HTML is for humans.

CommandHTML output
fanout planSummary cards (+install / −remove / =keep), per-adapter bucket sections, color-coded item lists
fanout applySnapshot id, installed/removed/failed sections, full install logs per bucket
fanout statePer-bucket ownership counts, owned-item lists, snapshot history table
fanout state diffDrift cards, per-bucket off-manifest installs and missing items, with rationale
fanout verifyPassed/failed/total counts, per-check status with stderr capture
fanout rollbackRe-installed and removed bucket sections, log capture
fanout ai "..."Generated manifest preview with accept status
fanout claude "..." "..."Aggregated per-worker output in collapsible sections

Two opt-outs:

  • --no-open — write the HTML, don't open the browser.
  • --no-html — terminal output only.

Markdown stopped scaling at ~100 lines. HTML scales. The chance you actually read your plan before applying it is much higher when it has color, summary cards, and grouping that text cannot do.

IX.What v3 does not do (yet)

  • Linux apt/dnf adapters. macOS-first; Linux is a v3.5 extension — straightforward, one more adapter file.
  • Windows. Indefinitely deferred. Brew is the design center.
  • Distributed apply over SSH. Multi-machine convergence (one tmux window per remote box) is v4.
  • Drift daemon. Opt-in launchd job that runs state diff hourly and notifies. v4.
  • Encrypted secrets. 1Password CLI integration for tokens at apply time. v5.
  • Module marketplace. Shareable .fanout modules. v5.
  • True parallelism in apply. v3 runs adapters sequentially; v4 will dispatch via tmux panes (the infra already exists in workers.py from fanout's prior life).
