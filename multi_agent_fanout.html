fanout — multi-agent dispatch

N parallel Claude agents in tmux, from one bash command

A typing-test backend in one terminal. A frontend in the next. A schema in a third. All running at the same time, prompted by you (or by an orchestrator Claude), all visible side by side in tmux. That is the entire tool. No config files, no installer, no state directory. Clone the repo, symlink one script, type fanout claude 4 "...", and four agents are working in parallel.

I.Use

Two commands

fanout claude N "prompt"        # N tmux panes, all running the same prompt
fanout claude N "p1" "p2" "p3"  # N panes, one prompt each (count must match)
fanout plan "task description"  # orchestrator-Claude decides N + prompts, then launches

Same prompt across N panes

fanout claude 2 "build a backend for a typing test app that will take tests and submit them"

Two panes spawn in a detached tmux session. Each runs claude -p with that prompt. You attach (tmux attach -t fanout_xxx) to watch live, or wait for the orchestrator to print aggregated output at the end.

Different prompts per pane

fanout claude 3 "design the schema" "write the API" "write the frontend"

Let Claude decide

fanout plan "build a typing-test app with Go backend, Postgres, and a React frontend"

An orchestrator Claude reads your task, decides how many agents make sense, writes one self-contained prompt per agent, and then launches them. Returns JSON {n, rationale, tasks} internally; you see the rationale + per-agent prompt previews before the panes spawn.

Add --dry-run to see the plan without launching.

II.Paste a repo, or any context

Every prompt can be prefixed with file content from a path on disk:

fanout claude 4 "audit each module for security issues" --repo ~/work/my-api
fanout plan "refactor this codebase to add caching" --repo .

The repo packer walks the directory, skips .git / node_modules / build dirs / binaries / oversized files, and dumps the contents into a <repo>...</repo> XML block at the top of every prompt. Capped at ~400 KB total so it stays in context.

Or pipe arbitrary text via stdin:

git diff HEAD~5 | fanout plan "review this diff for regressions" --paste
cat docs/spec.md | fanout claude 2 "implement this two different ways" --paste

Combine: --repo and --paste both wrap the prompt with context the agent reads before doing the task.

III.How it works

Each pane runs claude -p "<prompt>" | tee W{i}.out; touch W{i}.done; sleep 86400. The orchestrator polls the .done sentinels, collects outputs, prints them aggregated, then kills the session (unless --keep-tmux).

If tmux isn't on PATH, fanout falls back to asyncio.gather — same N parallel calls, output prints sequentially in your current terminal.

IV.Flags

Both claude and plan accept:

FlagEffect
--repo PATHPack repo contents (text files only, prune dirs/binaries) as preamble in every prompt.
--pasteRead stdin; prepend to every prompt.
--no-tmuxUse asyncio.gather instead of tmux. Output prints sequentially.
--keep-tmuxDon't kill the tmux session after panes finish.
--timeout SECONDSPer-pane timeout. Default 600.

plan also:

  • -n N — hint to the orchestrator on N (still its decision).
  • --dry-run — print the orchestrator's plan; don't launch panes.

V.Why this shape

Most multi-agent tooling is a framework. This is a bash command.

  • No install. Clone, symlink, done. No package manager. No pyyaml. No ~/.fanout. Five source files, all stdlib.
  • Two subcommands. claude is the dispatch primitive. plan is the orchestrator on top of it. You can read both in fifteen minutes.
  • tmux as the UI. Panes are the right abstraction for "N things happening at once that I want to watch". The framework layer is the operating system. We're not reinventing it.
  • Prompts are self-contained. No worker reads another's output. No graph. No DAG. No message passing. If your task can't be parallelised, the orchestrator returns N=1 and you get one agent. Simple shape, simple guarantees.
  • Repo context built in. --repo is the killer flag. Audit my whole codebase from N angles, in parallel, with the code already in each agent's context.

What this tool deliberately is not: an agent framework, a workflow engine, a config management system, a build tool. It is one bash command that puts N Claudes side by side and gets out of your way.
